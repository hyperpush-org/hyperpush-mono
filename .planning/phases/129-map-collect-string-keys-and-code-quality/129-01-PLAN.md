<objective>
Fix Map.collect to correctly use string-key variant when collecting from a zip iterator of string-keyed pairs.

Purpose: Currently, `string_keys |> Iter.zip(values) |> Map.collect()` silently uses the integer-key collect variant, making `Map.get(m, "key")` return wrong results (0 instead of the actual value). The `pipe_chain_has_string_keys` heuristic only recognizes `List<(String,V)>` or `Map<String,V>` source types, but not the `Iter.zip` pattern.

Output: Fixed lower_pipe_expr that checks the pipe's resolved result type to detect string-keyed maps, plus a new E2E test verifying the zip-then-collect pattern works.
</objective>

/// Check if a Ty is Map<String,V> or List<(String,V)>
fn ty_has_string_map_keys(ty: &Ty) -> bool {
    match ty {
        Ty::App(con, args) => {
            if let Ty::Con(ref tc) = **con {
                match tc.name.as_str() {
                    "Map" if !args.is_empty() => args[0] == Ty::string(),
                    "List" if !args.is_empty() => {
                        if let Ty::Tuple(elems) = &args[0] {
                            !elems.is_empty() && elems[0] == Ty::string()
                        } else { false }
                    }
                    _ => false,
                }
            } else { false }
        }
        _ => false,
    }

**Root cause of the bug:**
- `Map.collect` is typed `forall K V. Fn(Ptr) -> Map<K,V>` in builtins.rs
- `Iter.zip` is typed `Fn(Ptr, Ptr) -> Ptr` — result is opaque Ptr, no element type
- So `pipe_chain_has_string_keys` can never detect string keys from a zip chain by inspecting the source type
- BUT: if `Map.get(m, "str_key")` is called after the collect, the union-find unifies K=String
  in the SAME resolved types map that lower.rs uses
- Therefore: `self.types.get(&pipe.syntax().text_range())` for `|> Map.collect()` pipe
  returns `Map<String, V>` AFTER the final resolve pass IF the result was used with a string key

**Fix strategy:**
In `lower_pipe_expr`, when `mesh_map_collect` is detected, add a FIRST check:
look up `self.types.get(&pipe.syntax().text_range())` — if that Ty is `Map<String, V>`,
use the string-key variant. Only fall back to `pipe_chain_has_string_keys` if the result
type is unresolved or not string-keyed.

This handles:
1. `zip + Map.get("string")` pattern ← result type resolves to Map<String,V>
2. `Map.to_list roundtrip` ← pipe chain source is List<(String,V)> (existing heuristic)
